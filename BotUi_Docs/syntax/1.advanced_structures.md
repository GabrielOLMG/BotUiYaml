# Dynamic Expansion: Loops and Imports

## Overview

To keep YAML files maintainable and scalable, BotUi provides two structural commands: FOR_EACH and IMPORT_ACTIONS. These are not browser actions; they are instructions for the Expander to generate a sequence of actions dynamically before the bot starts.

---

## 1. ``FOR_EACH`` (Iterations)

The FOR_EACH command allows you to repeat a block of steps for every item in a list. This is essential for stress testing scenarios where you need to perform the same flow with different data inputs.

Why use it?

- Massive Data Input: Fill out forms or create records using a list of 100+ users.
- Cleaner YAML: Instead of writing 1,000 steps, you write 5 steps inside a loop.

Usage:

- loop_var: The name of the variable to represent the current item (e.g., item).
- items: A list of dictionaries or a reference to a variable (e.g., $USER_LIST).
- steps: The list of actions to be repeated.

```yaml
- action: FOR_EACH
  loop_var: user_info
  items: # This list could be contained within the JSON file: "{{LOAD_FILE.data/users_info.json}}"
    - {username: Gabriel, age: 28}
    - {username: Raphael, age: 31}
  steps: # Actions that will be repeated for each element in items
    - action: FIND
      object_type: TEXT
      text: "username"
      interaction:
        type: CLICK  
    - action: WRITE
      text: "{user_info.username}"
    - action: FIND
      object_type: TEXT
      text: "age"
      interaction:
        type: CLICK  
    - action: WRITE
      text: "{user_info.age}"
```

In other words, instead of repeating these four steps N times, they are wrapped within a for-each loop, allowing the same process to be applied to all elements contained in items.

## 1. ``IMPORT_ACTIONS`` (Modularization)

The **IMPORT_ACTIONS** command allows you to include steps from an external YAML file. This promotes reusability across different bot projects.

Why use it?

- Common Flows: Create a **login_flow.yaml** and import it into every bot you build.
- Organization: Split giant YAML files into smaller, manageable chunks (e.g., setup.yaml, teardown.yaml).

Usage

- path: The relative path to the YAML file containing a list of steps.


Main YAML:

```yaml
pipelines:
  PipelineExample:
    url: $URL
    steps:
      - action: IMPORT_ACTIONS
        path: "modules/login.yaml"
```

modules/login.yaml

```yaml
- action: FIND
  object_type: TEXT
  text: "E-mail"
  interaction:
    type: CLICK  
    offset:
      y: 30
  debug: true

- action: WRITE
  text: $USERNAME

- action: FIND
  object_type: TEXT
  text: "Password"
  interaction:
    type: CLICK  
    offset:
      y: 30

- action: WRITE
  name: "write_password"
  text: $PASSWORD

- action: FIND
  object_type: TEXT
  text: "Login"
  debug: true
  interaction:
    type: CLICK  
  position: 1
```

Note that the file containing the modularized actions is structured as a list of actions, rather than a pipeline format.

The goal of this approach is to prevent the main bot YAML file from becoming overly large, while also making it easier to apply targeted fixes.
